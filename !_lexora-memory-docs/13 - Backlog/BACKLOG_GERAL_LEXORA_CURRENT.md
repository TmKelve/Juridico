- **Ordenação por criticidade** — não por data de criação ou descoberta.
- **Decisões antes de correções** — quando uma decisão é pré-requisito para uma correção, ela aparece primeiro na tabela, mesmo que a correção seja mais urgente operacionalmente.

## 2. Regras de Priorização

| Prioridade | Critério |
|---|---|
| **P0** | Risco crítico em produção, segurança, dados ou bloqueio forte de operação |
| **P1** | Risco alto, decisão importante ou bloqueio de documentação confiável |
| **P2** | Melhoria relevante, dívida técnica ou organização importante |
| **P3** | Limpeza, refinamento ou baixa urgência |

> [!warning] P0 não pode ficar em aberto indefinidamente
> Itens P0 devem ter responsável designado e prazo definido. Se permanecerem em aberto após validação, devem ser escalados.

---

## 3. Ordem Oficial da Tabela

A tabela deve ser ordenada respeitando **sempre** esta sequência de critérios:

1. **Prioridade:** P0 → P1 → P2 → P3
2. **Tipo** (dentro da mesma prioridade):
   1. Decisão
   2. Validação
   3. Correção técnica
   4. Segurança
   5. Governança
   6. Documentação
   7. Arquivamento
   8. Limpeza
   9. Melhoria
   10. Refatoração
   11. Design System
   12. Organização
   13. Implementação
3. **Dependência** — itens que desbloqueiam outros vêm antes
4. **Área**
5. **Origem** (KB, ADR, validação manual, etc.)

> [!note] Princípio operacional
> O backlog representa **ordem de atuação**, não ordem de descoberta. Uma decisão P1 que desbloqueia três correções P2 deve aparecer antes dessas correções, mesmo que elas sejam tecnicamente mais simples.
